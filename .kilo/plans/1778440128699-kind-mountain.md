# Plan: Directed Visual Notes

## Objectif
Introduire une couche `DirectedVisualNote` lambda-first qui laisse les visual notes écrire leur chorégraphie custom, tout en centralisant les pièges de timing, ownership par track, forward-crossing, rewind/seek et mutation safe.

Le premier livrable doit rester incrémental : ajouter l’infrastructure générique et migrer `SeaponyBgVisualNote` comme preuve de concept. `SeaponyVisualNote` et `SeeSawVisualNote` ne doivent pas être réécrites d’un bloc.

## Constats Codebase
- `VisualNote` et `VisualNoteManager<T>` vivent dans `MonogameLibs/RhythmConductor`; ils fournissent déjà le state timing déterministe (`Progress`, `UnclampedProgress`, `PostHitProgress`, spawn/despawn). La nouvelle couche peut dériver de `VisualNote` sans modifier le manager.
- `SeaponyBgVisualNote` est le meilleur premier candidat : elle fait seulement fenêtre post-hit, `Func<bool> _canApplyState`, interpolation background.
- `SeaponyVisualNote` concentre les patterns à extraire ensuite : `_lastSongPosition`, SFX forward-crossing, rewind reset, `_wasControlling`, tap-tap pose memory, `Func<bool> _canApplyState`.
- `SeeSawScene` n’utilise plus `SeeSawVisualNote`; elle pilote un `SeeSawDirector` spécialisé. `SeeSawVisualNote` reste présent mais semble legacy/non référencé. Ne pas baser le premier incrément dessus.
- `SeeSawDirector` a déjà un runtime spécialisé avec crossing sets et rewind reset. La couche générique pourra plus tard s’inspirer de ses idées, mais ne doit pas remplacer ce director dans la première migration.

## Architecture À Ajouter
Créer les nouveaux fichiers sous `Gameplay/Visual Notes/Directed/` pour éviter de toucher la dépendance `RhythmConductor` dans un premier temps.

### `VisualRuntime`
Responsabilités : registre de tracks et arbitrage d’écriture.

API cible initiale :
```csharp
public sealed class VisualRuntime
{
    public VisualTrack<T> RegisterTrack<T>(string id, T target);
    public VisualTrack Track(string id);
    public bool CanWrite(string trackId, Note note, double songPosition);
    public bool TryRead<T>(string trackId, out T target);
    public void SetDriver(string trackId, Note driver);
    public void ClearDrivers();
}
```

Décisions initiales :
- Commencer avec un driver manuel par track (`SetDriver(trackId, note)`) pour remplacer directement les `Func<bool> canApplyState` existants.
- Garder l’extension vers policies plus tard via une interface `IVisualDriverPolicy`, mais ne pas la sur-développer avant la première migration.
- `CanWrite` doit retourner `false` si la track n’existe pas ou si le driver ne correspond pas, afin d’éviter les mutations silencieusement globales.

### `VisualTrack`
Responsabilités : stocker l’objet cible, son id, son driver courant, et éventuellement une policy future.

API cible initiale :
```csharp
public sealed class VisualTrack
{
    public string Id { get; }
    public object Target { get; }
    public Note DriverNote { get; private set; }
    public void SetDriver(Note note);
}

public sealed class VisualTrack<T> : VisualTrack
{
    public new T Target { get; }
}
```

### `VisualContext`
Responsabilités : contexte safe passé aux lambdas.

API cible initiale :
```csharp
public sealed class VisualContext
{
    public Note Note { get; }
    public double SongPosition { get; }
    public double LastSongPosition { get; }
    public double ApproachDuration { get; }
    public double DespawnDelay { get; }
    public float NoteProgress { get; }
    public float UnclampedNoteProgress { get; }
    public float PostHitProgress { get; }
    public bool HasRewound { get; }
    public bool IsBeforeApproach { get; }
    public bool IsAtOrAfterHit { get; }

    public bool CanWrite(string trackId);
    public bool TryRead<T>(string trackId, out T target);
    public T Read<T>(string trackId);
    public void Mutate<T>(string trackId, Action<T> mutation);
    public bool ForwardCrossed(string eventId, double songTime);
    public bool ForwardCrossedProgress(string eventId, float noteProgress);
    public void ForceAnimation(string animationTrackId, string stateName, string reenterViaState = null);
    public void PlaySfxOnForwardCross(string eventId, double songTime, Scene scene, string path, float volume);
    public void SpawnOnForwardCross(string eventId, double songTime, Action spawn);
}
```

Décisions initiales :
- `Mutate` doit vérifier deux choses : la lambda courante a déclaré posséder la track via `.Owns(...)`, et le runtime autorise la note courante à écrire sur cette track.
- `ForceAnimation` doit lire une track `AnimationStateMachine` et appeler `RhythmVisualUtils.ForceAnimationState`.
- `PlaySfxOnForwardCross` doit rester opt-in et prendre `Scene scene` au début, pour ne pas imposer que `VisualContext` connaisse toutes les scènes.
- `ForwardCrossed` ne doit pas tirer au premier sample (`LastSongPosition` NaN), ni en rewind, ni en seek direct après le seuil. Il doit redevenir disponible naturellement après rewind puis recroisement.

### `VisualEventGate`
Responsabilités : calculs forward-only sans flags par visual note.

API cible initiale :
```csharp
public sealed class VisualEventGate
{
    public bool ForwardCrossed(string eventId, double lastSongPosition, double songPosition, double eventSongPosition, bool hasRewound);
    public bool ForwardCrossedProgress(string eventId, float lastProgress, float progress, float eventProgress, bool hasRewound);
}
```

Règle : `last < threshold && current >= threshold && !hasRewound && !double.IsNaN(last)`.

### `PhaseContext`
Responsabilités : progression locale/globale et helpers de crossing de phase.

API cible initiale :
```csharp
public sealed class PhaseContext
{
    public string Id { get; }
    public float GlobalStart { get; }
    public float GlobalEnd { get; }
    public float GlobalProgress { get; }
    public float LocalProgress { get; }
    public float UnclampedLocalProgress { get; }
    public bool IsActive { get; }
    public bool WasActive { get; }

    public bool JustEnteredForward(string eventId);
    public bool JustExitedForward(string eventId);
    public bool CrossedLocalForward(string eventId, float localProgress);
    public bool CrossedGlobalForward(string eventId, float globalProgress);
}
```

Décisions :
- Garder `Local` et `Global` explicites.
- `WasActive` se calcule depuis `LastSongPosition` et la progression précédente.
- `JustEnteredForward` correspond au crossing du `GlobalStart`.
- `JustExitedForward` correspond au crossing du `GlobalEnd`.

### `VisualTimeline`, `VisualPhase`, `VisualStableBlock`
Responsabilités : déclarer des blocs lambda-first et les sampler.

API cible initiale :
```csharp
public sealed class VisualTimeline
{
    public VisualPhase Phase(string id, float startProgress, float endProgress);
    public VisualPhase DuringApproach(string id);
    public VisualPhase AfterHit(string id);
    public VisualPhase AfterHitUntilDespawn(string id);
    public VisualStableBlock StableBefore(string id = "stable_before");
    public VisualStableBlock StableAfter(string id = "stable_after");
    public void Sample(VisualContext context);
}

public sealed class VisualPhase
{
    public VisualPhase Owns(params string[] trackIds);
    public VisualPhase Do(Action<VisualContext, PhaseContext> action);
    public VisualPhase DoOwned<T>(string trackId, Action<VisualContext, PhaseContext, T> action);
}

public sealed class VisualStableBlock
{
    public VisualStableBlock Owns(params string[] trackIds);
    public VisualStableBlock Do(Action<VisualContext> action);
}
```

Sampling :
- `Phase(start,end)` est basé sur la progression d’approche globale `[0..1]` de la note.
- `DuringApproach` équivaut à `[0,1]` avant hit.
- `AfterHitUntilDespawn` utilise `ctx.PostHitProgress` et la fenêtre `[Note.EndSongPosition, DespawnSongPosition]`.
- `StableBefore` s’active avant le début d’approche.
- `StableAfter` s’active à partir de la fin de fenêtre visuelle ou, si utile pour les migrations, peut être laissé non utilisé jusqu’à un cas concret. Ne pas forcer son usage tant que SeaPony background n’en a pas besoin.

### `DirectedVisualNote`
Responsabilités : base au-dessus de `VisualNote` qui construit une timeline et injecte le contexte.

API cible :
```csharp
public abstract class DirectedVisualNote : VisualNote
{
    protected DirectedVisualNote(Note note, VisualRuntime runtime, double approachDuration, double despawnDelay = 0)
        : base(note, approachDuration, despawnDelay)
    {
        Runtime = runtime ?? throw new ArgumentNullException(nameof(runtime));
        Timeline = new VisualTimeline();
        Build(Timeline);
    }

    protected VisualRuntime Runtime { get; }
    protected VisualTimeline Timeline { get; }
    protected abstract void Build(VisualTimeline timeline);
    public sealed override void Update(double currentSongPosition);
}
```

`Update` :
- Appeler `base.Update(currentSongPosition)` pour conserver `VisualNote.State`.
- Construire `VisualContext` avec `Note`, `currentSongPosition`, `_lastSongPosition`, `ApproachDuration`, `DespawnDelay`, `Runtime`, `State`.
- Sampler la timeline.
- Mettre à jour `_lastSongPosition` après sampling.

## Migration Incrémentale

### Étape 1 : Infrastructure sans migration lourde
- Ajouter les types `VisualRuntime`, `VisualTrack`, `VisualEventGate`, `VisualContext`, `PhaseContext`, `VisualTimeline`, `VisualPhase`, `VisualStableBlock`, `DirectedVisualNote`.
- Ne pas modifier `VisualNoteManager<T>`.
- Ne pas toucher `SeeSawDirector`.
- Ajouter des tests unitaires pour l’ownership, les crossings, les progressions de phase et le sampling minimal.

### Étape 2 : Migrer `SeaponyBgVisualNote`
Deux options acceptables :
- Remplacer directement la classe existante par une version qui dérive de `DirectedVisualNote`.
- Ou créer `SeaponyBgDirectedVisualNote`, puis changer la factory de `SeaPonyParade`.

Préférence : remplacer directement `SeaponyBgVisualNote`, car la classe est petite et déjà isolée.

Changements côté `SeaPonyParade` :
- Ajouter un champ `_visualRuntime`.
- Dans `createVisualNotes`, créer/réinitialiser le runtime et enregistrer `background` :
```csharp
_visualRuntime.RegisterTrack("background", _infiniteScrollBg);
```
- Dans `updateSeaPonies`, après `_drivingBackgroundNote = findDrivingBackgroundNote(songPos);`, appeler :
```csharp
_visualRuntime.SetDriver("background", _drivingBackgroundNote);
```
- Modifier la factory background :
```csharp
return new SeaponyBgVisualNote(note, _visualRuntime, crotchet, getBackgroundScrollDestinationBeat(note), getBackgroundScrollDuration(action, note));
```

Nouvelle logique `SeaponyBgVisualNote` :
```csharp
protected override void Build(VisualTimeline t)
{
    t.AfterHitUntilDespawn("background_scroll")
        .Owns("background")
        .DoOwned<InfiniteScrollBackground>("background", (ctx, phase, bg) =>
        {
            if (!SeaponyNoteCodec.TryReadAction(ctx.Note?.AdditionnalData, out _))
                return;

            float p = ctx.PostHitProgress;
            float interpolated = Interpolation.EaseOutQuart(p);
            bg.Progression = Single.Lerp(_backgroundScrollDestinationBeat - 1, _backgroundScrollDestinationBeat, interpolated);
        });
}
```

### Étape 3 : Tests et validation
Ajouter un fichier de tests, par exemple `MLP_RiM.Tests/DirectedVisualNoteTests.cs`.

Cas minimaux :
- `VisualRuntime` applique une mutation seulement si la track existe, est déclarée owned, et le driver est la note courante.
- `ForwardCrossed` est faux au premier sample, vrai sur crossing forward, faux sans crossing, faux en rewind, puis vrai après recroisement.
- `PhaseContext` calcule `LocalProgress`, `UnclampedLocalProgress`, `IsActive`, `WasActive`, `JustEnteredForward` et `CrossedLocalForward` correctement.
- `AfterHitUntilDespawn` sample la phase seulement dans la fenêtre post-hit.
- `SeaponyBgVisualNote` met à jour un faux background uniquement quand `background` est drivé par la note.

Validation manuelle/commande :
- `dotnet test MLP_RiM.Tests/MLP_RiM.Tests.csproj`
- `dotnet build MLP_RiM.csproj`

### Étape 4 : Migrer des morceaux de `SeaponyVisualNote` plus tard
Après la preuve background :
- Remplacer `playSfxOnForwardCross` par `ctx.PlaySfxOnForwardCross` pour Swim/Roll/TapTap.
- Remplacer progressivement `_canApplyState` par tracks `seapony_i` et `seapony_anim_i`.
- Garder `_wasControlling`, `_lastTapTapHitsPassed`, `_lastTapTapReactionState` au début. Ne pas introduire de state bag tant qu’un vrai besoin n’est pas prouvé.
- Convertir les actions une par une (`Swim`, puis `Roll`, puis `TapTap`, puis `Leave`).

### Étape 5 : Policies de drivers
Après que le runtime manuel ait remplacé au moins background :
- Ajouter `IVisualDriverPolicy` seulement si la répétition côté scène devient gênante.
- Encapsuler `findDrivingBackgroundNote` et `findDrivingSeaPonyNote` dans des policies dédiées.
- L’objectif est de déplacer la décision de driver vers le runtime, pas de changer la logique de priorité en même temps.

### Étape 6 : SeeSaw
Ne pas migrer `SeeSawDirector` dans ce chantier initial.

Décision recommandée :
- Traiter `SeeSawVisualNote` comme legacy tant qu’aucune scène ne l’utilise.
- Si une migration SeeSaw est voulue plus tard, comparer d’abord `SeeSawDirector` à `DirectedVisualNote`. Le director possède déjà des concepts proches : segments actifs, sets de crossings, rewind reset, state stable et controllers par ressource.
- Éviter de forcer SeeSaw dans `VisualNoteManager` si le director actuel est plus adapté à sa timeline multi-acteurs.

## Risques
- `StableAfter` peut être ambigu avec `AfterHitUntilDespawn`; ne pas le sur-spécifier avant un cas concret.
- Les animations one-shot et les animations déterministes ne doivent pas être confondues. Les SFX/spawns utilisent `ForwardCrossed`; les poses/animations nécessaires après seek doivent être réappliquées par état stable ou état discret déterministe.
- Les tracks manquantes doivent échouer de façon sûre. Pour la première version, préférer no-op côté `Mutate` et tests explicites plutôt qu’une exception en runtime de jeu.
- Une migration complète de `SeaponyVisualNote` peut changer des subtilités de takeover tap-tap/roll. Elle doit être fractionnée.

## Critère De Succès Du Premier Incrément
- La couche `DirectedVisualNote` existe et est utilisable en lambda-first.
- `SeaponyBgVisualNote` ne contient plus de `Func<bool> canApplyState` et passe par `ctx.Mutate`/ownership track.
- Les tests couvrent ownership, crossing forward/rewind et phases.
- `VisualNoteManager<T>` et `SeeSawDirector` restent compatibles et inchangés.
