# Plan: Dedicated Same-Variant Hit Windows

## Goal
Allow notes with the same editor definition and same variant to be placed closer together than they would be allowed against other notes/variants, while still keeping the normal hit window for blocking different notes. Render a second hit-window bar above the existing red hit-window bar to visualize the same-variant placement window. If no dedicated same-variant window is configured, it defaults to the normal hit window.

## Current Findings
- Placement blocking is centralized in `Elements/Editor/BeatmapEditorDocument.cs` via `FindBlockingNote`, `PlacementsBlockEachOther`, `GetBlockingStart`, and `GetBlockingEnd`.
- Rendering of note occupancy and hit windows is in `Elements/Editor/BeatmapEditorElement.cs` around `DrawTimeline`, currently drawing one red hit-window bar at `area.Y + 118`.
- Timing abstraction is `IEditorNoteTiming`, implemented by `FixedEditorNoteTiming`, `SeeSawEditorNoteTiming`, and `SeaponyParadeEditorNoteTiming`.
- `SeaponyParadeEditorNoteTiming` currently stretches roll hit windows based on roll sequence/end timing, which is correct for blocking other notes but too wide for placing another roll one beat later.
- Definition configuration is built through `EditorNoteDefinitionBuilder`, with base values stored in `EditorNoteDefinition`.

## Implementation Steps
1. Extend `IEditorNoteTiming` with two new methods:
   - `GetSameVariantHitWindowStart(EditorNoteDefinition definition, EditorNoteTimingContext context)`
   - `GetSameVariantHitWindowEnd(EditorNoteDefinition definition, EditorNoteTimingContext context)`

2. Update timing implementations:
   - `FixedEditorNoteTiming`: return the normal hit-window start/end by default.
   - `SeeSawEditorNoteTiming`: return the normal hit-window start/end by default unless a specific need appears.
   - `SeaponyParadeEditorNoteTiming`: for roll notes, return a same-variant window from `SongPosition` to `SongPosition + Crotchet`; for other variants, return the normal hit window. This lets roll -> roll at one beat interval touch the boundary without overlap.

3. Extend `EditorNoteDefinition` with wrapper methods mirroring the existing hit-window helpers:
   - `GetSameVariantHitWindowStart(...)`
   - `GetSameVariantHitWindowEnd(...)`
   Include overloads needed by `BeatmapEditorDocument` and renderer, especially `(songPosition, crotchet, variantIndex, getRelativeNote)`.

4. Add optional builder support for fixed same-variant windows:
   - Add nullable fields in `EditorNoteDefinitionBuilder` such as `_sameVariantHitWindowBeforeBeats` and `_sameVariantHitWindowAfterBeats`.
   - Add a method like `SameVariantHitWindow(double beforeBeats, double afterBeats)`.
   - Store these optional values in `EditorNoteDefinition`.
   - `FixedEditorNoteTiming` should use these values when present, otherwise fall back to `HitWindowBeforeBeats` / `HitWindowAfterBeats`.
   This satisfies the “by default if not specified, same as normal hit window” requirement for simple note definitions.

5. Update placement blocking in `BeatmapEditorDocument`:
   - Determine whether placed and existing notes are the same editor definition and same variant.
   - If same definition + same variant, compare using same-variant hit windows for both notes.
   - Otherwise, compare using the normal hit windows.
   - Keep `HitWindowEpsilonSeconds` boundary behavior so notes exactly one beat apart are allowed when one ends exactly where the next starts.
   - Apply the same logic both for placement against existing chart notes and generated placements against each other.

6. Add helper methods in `BeatmapEditorDocument`:
   - `AreSameDefinitionAndVariant(EditorNoteDefinition aDefinition, ChartNote aNote, EditorNoteDefinition bDefinition, ChartNote bNote)`.
   - `GetBlockingStart(..., bool sameVariantWindow)` and `GetBlockingEnd(..., bool sameVariantWindow)` or equivalent to avoid duplicating the contextual SeeSaw special cases.
   - Keep existing SeeSaw boundary exception intact unless same-variant logic cleanly supersedes it.

7. Update rendering in `BeatmapEditorElement.DrawTimeline`:
   - Continue drawing the existing red normal hit-window bar.
   - Draw a second same-variant hit-window bar above it, e.g. at `area.Y + 106` or similar, using a distinct color such as orange/yellow.
   - Compute it from `_document.GetContextualSameVariantHitWindowStart(note, definition)` and `_document.GetContextualSameVariantHitWindowEnd(note, definition)`.
   - If same-variant window equals normal hit window, rendering it is still acceptable per request; alternatively draw it slightly offset so both remain visible.

8. Add contextual accessors in `BeatmapEditorDocument`:
   - `GetContextualSameVariantHitWindowStart(ChartNote note, EditorNoteDefinition definition)`
   - `GetContextualSameVariantHitWindowEnd(ChartNote note, EditorNoteDefinition definition)`
   These should mirror normal contextual hit-window methods and pass variant index plus relative note getter.

9. Configure Seapony roll behavior:
   - In `SeaponyParadeEditorNoteTiming`, normal hit window remains the current roll timing for interactions with other notes.
   - Same-variant hit window for `RollAction` should be exactly one crotchet after the roll start (`SongPosition` to `SongPosition + Crotchet`).
   - This should allow placing consecutive roll variants every beat while still blocking swim/tap/other notes over the full roll timing.

10. Verify:
   - Run `dotnet build`.
   - Manually verify a roll can be placed one crotchet after another roll.
   - Verify a different variant/note is still blocked by the normal roll hit window.
   - Verify the timeline shows two bars: normal hit window and same-variant hit window.

## Notes
- This plan avoids changing chart serialization.
- Same-variant behavior is based on `EditorNoteDefinitions.FindVariantIndex`, so Seapony `Roll` is distinguished from `Swim` and `Tap Tap`.
- Existing definitions will behave the same unless they configure or override same-variant timing.
