# Plan: timeline beat-first et tempo map projetee

## Objectif

Faire de la position en beats la source de verite du chart, et releguer les secondes a une projection runtime via `ChartTempoMap`.

Invariant cible:

```csharp
editorX = beat * pixelsPerBeat;
runtimeSeconds = tempoMap.BeatToSeconds(beat);
currentBeat = tempoMap.SecondsToBeat(audioSongPosition);
```

Ce plan est separe en phases pour eviter un big bang. Les anciennes API secondes restent temporairement comme wrappers de compatibilite, mais le nouveau code editeur doit appeler les API beat-first.

## Constat codebase

- Les types XML `Chart`, `ChartNote`, `ChartEffect` sont dans `..\..\MonogameLibs\RhythmConductor\RhythmConductor\Note\Chart.cs`, pas dans le projet jeu.
- `Gameplay/ChartTempoMap.cs` est architecturalement au bon endroit, mais ses segments sont encore construits depuis `SongPosition` et exposent `GetBeatAt` / `GetSongPositionAtBeat` autour des secondes.
- `Elements/Editor/BeatmapEditorDocument.cs` normalise, trie, place, deplace et bloque les notes avec `SongPosition`; les holds sont recalcules par `definition.HoldBeats * GetCrotchetAt(note.SongPosition)`.
- `Elements/Editor/BeatmapEditorElement.cs` dessine la timeline avec `TimeToX`, `XToSongPosition`, `_manualSongPosition`, `_visibleBeforeSeconds`, `GetTempoSegments`, et des ancres `section_offset`.
- `Gameplay/BeatmapPlayer.cs` applique les effets BPM en appelant `Conductor.SetBpm`, puis met a jour `ChartPlayer` et les visuels en secondes.
- `Gameplay/SeeSaw/SeeSawRuntime.cs` utilise deja des beats pour la logique See-Saw, mais trie et compare encore souvent des `SongPosition`, et le director interpole plusieurs parties via secondes.
- `Elements/Editor/EditorRhythmInputVisualNote.cs` utilise `crotchet * 2` comme duree d’approche en secondes.
- Il n’existe pas de projet de tests actuellement.

## Phase 0: tests et garde-fous

- Ajouter un projet de tests `.NET 9` sous le repo, par exemple `MLP_RiM.Tests/MLP_RiM.Tests.csproj`, avec references vers `MLP_RiM.csproj` et `RhythmConductor.csproj`.
- Utiliser des tests unitaires purement timing, sans scene MonoGame ni audio BASS.
- Ajouter les tests de base avant le refactor de l’editeur:
- `BeatToSeconds(SecondsToBeat(t)) ~= t`.
- `SecondsToBeat(BeatToSeconds(b)) ~= b`.
- `GetBpmAtBeat` et `GetBpmAtSeconds` avant, au point exact, et apres chaque effet BPM.
- Changer le BPM d’un effet ne change aucune `BeatPosition` de note.
- Une note a beat 40, un BPM change a beat 41, des notes a beats 42 et 46: modifier 120 -> 180 ne change pas `EditorX`, `BeatPosition`, `HoldBeats`, mais change les secondes projetees apres beat 41.
- Ajouter des helpers d’assertion `NearlyEqual` et, pour See-Saw, comparer les beats via ticks.

## Phase 2: `ChartTempoMap` beat-first

- Refactorer `Gameplay/ChartTempoMap.cs` autour de segments `StartBeat`, `StartSeconds`, `Bpm`.
- Ajouter les API canoniques:

```csharp
public double BeatToSeconds(double beat);
public double SecondsToBeat(double seconds);
public double GetBpmAtBeat(double beat);
public double GetBpmAtSeconds(double seconds);
public double GetSecondsPerBeatAtBeat(double beat);
```

- Garder temporairement les alias legacy:

```csharp
public double GetBeatAt(double songPosition) => SecondsToBeat(songPosition);
public double GetSongPositionAtBeat(double beat) => BeatToSeconds(beat);
public double GetBpmAt(double songPosition) => GetBpmAtSeconds(songPosition);
public double GetCrotchetAt(double songPosition) => 60.0 / GetBpmAtSeconds(songPosition);
```

- Construire le premier segment avec `StartBeat = 0.0`, `StartSeconds = chart.Offset`, `Bpm = chart.BPM`.
- Construire les segments BPM en triant les effets par beat, pas par secondes.
- Pour les charts v1 sans `BeatPosition`, creer une tempo map legacy interne qui reproduit l’ancien tri par `effect.SongPosition`, puis convertir les effets en beats via `legacyMap.SecondsToBeat(effect.SongPosition)`.
- Ignorer `section_offset` dans la nouvelle projection beat -> secondes.
- Conserver une methode editor temporaire pour afficher un warning si un BPM change a `section_offset != 0`.
- Ajouter une API de segments beat-based pour l’editeur, par exemple `GetTempoSegmentsByBeat(double startBeat, double endBeat)`, sans ancre locale.
- Traiter les BPM invalides ou <= 0 comme l’existant: fallback a 100 pour base chart, fallback a `0.6` secondes par beat si necessaire.
- Definir une politique deterministic pour plusieurs BPM au meme beat: trier par beat puis ordre original, et laisser le dernier effet au meme beat gagner ou remplacer le segment precedent.

- Pour See-Saw, continuer a utiliser le compilateur, mais lui fournir des beats via `GetNoteBeat`.
- Remplacer `HitWindowEpsilonSeconds` par `BeatEpsilon`, puis par `BeatTick` pour les comparaisons See-Saw sensibles.
- Garder des wrappers `GetContextualStartSeconds` etc. uniquement pour affichage ou compatibilite temporaire.

## Phase 4: placement, snap et timeline editeur en beats

- Dans `BeatmapEditorElement`, remplacer progressivement:
- `_manualSongPosition` par `_manualBeatPosition`.
- `_visibleBeforeSeconds` / `_visibleAfterSeconds` par `_visibleBeforeBeats` / `_visibleAfterBeats`.
- `_dragPointerOffsetSeconds` par `_dragPointerOffsetBeats`.
- `CurrentSongPosition()` par `CurrentBeatPosition()`.
- `Seek(double songPosition)` par `SeekBeat(double beat)`.
- `XToSongPosition` / `TimeToX` par `XToBeat` / `BeatToX`.
- `ClampSongPosition` par `ClampBeat`, avec conversion vers secondes seulement pour l’audio.
- Implementer `SnapBeat`:

```csharp
private double SnapBeat(double beat)
{
    double step = 1.0 / GetEffectiveSnapDivisions();
    return Math.Round(beat / step, MidpointRounding.AwayFromZero) * step;
}
```

- Les fleches sans shift avancent/reculent d’un pas de snap en beats.
- Le drag note/effect appelle `MoveNoteToBeat` / `MoveEffectToBeat`.
- Le placement note/effect cree les objets a beat via les nouvelles API.
- La selection utilise une distance en beats, par exemple `1.0 / snapDivisions`, pas `crotchet / snapDivisions`.
- Le dessin de timeline devient:

```csharp
x = area.Left + (beat - windowStartBeat) / (windowEndBeat - windowStartBeat) * area.Width;
```

- Dessiner la grille par beats constants: beats entiers en lignes fortes, subdivisions par `1 / snapDivisions`, sans `section_offset` ni ancre BPM locale.
- Les marqueurs BPM sont a `GetEffectBeat(effect)`.
- Les notes, holds, fenetres d’occupation et hit windows utilisent les beats et gardent la meme largeur visuelle quand un BPM change.
- Le HUD affiche les deux valeurs: `BEAT`, `SEC`, `BPM`, mais toute interaction timeline doit rester beat-first.

## Phase 10: nettoyage et deprecation

- `GetBeatAt(double songPosition)`.
- `GetSongPositionAtBeat(double beat)` si le nom reste ambigu, ou le renommer vers `BeatToSeconds`.
- `GetCrotchetAt(double songPosition)` en faveur de `GetSecondsPerBeatAtBeat` / `GetBpmAtBeat`.
- `ApplyChartEffectsAt`.
- `Conductor.SongPositionInBeatsExact` ne doit plus etre lu par le jeu.
- Renommer `Offset` vers `OffsetSeconds` seulement si la migration XML le permet; sinon documenter que `Offset` reste legacy XML.
- Supprimer `section_offset` du panneau BPM apres periode de compatibilite.
- Supprimer les wrappers secondes de l’editeur quand tous les appels sont beat-first.

## Risques et decisions techniques

- `ChartPlayer`, `Note`, `VisualNote` et `VisibleNoteWindow` sont encore seconds-based dans `RhythmConductor`; la projection runtime est le chemin minimal pour eviter un refactor externe massif immediat.
- Les tests peuvent etre ralentis par la reference au projet MonoGame; il faudra garder les tests sur des classes timing pures et eviter d’instancier `BeatmapPlayer.Conductor`.
- Les champs `double?` XML peuvent necessiter des proprietes `Specified` pour garder les XML propres.
- `section_offset` ne doit plus influencer la grille, mais les charts v1 qui l’utilisent doivent afficher un warning pour ne pas masquer une difference d’interpretation.
- Les mutations BPM peuvent changer les `SongPosition` derives lors d’une projection ou d’une sauvegarde v2; l’invariant important est que `BeatPosition`, `HoldBeats` et `editorX` ne changent pas.
