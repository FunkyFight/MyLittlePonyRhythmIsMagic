# Plan: RhythmVisualUtils pour notes visuelles deterministes

## Objectif
Creer un fichier utilitaire reutilisable pour extraire de `SeeSawVisualNote.cs` les mecaniques generiques de synchronisation visuelle: fenetre temporelle deterministe, rewind, ownership guard, triggers one-shot, interpolation d'arcs et timing derive par proximite. Le refactor doit garder le comportement See-Saw identique tout en donnant une base propre aux prochaines VisualNotes.

## Contexte observe
- `SeeSawVisualNote` est une `VisualNote` qui controle des `GameObject` possedes par la scene.
- Plusieurs VisualNotes peuvent etre actives simultanement via `VisualNoteManager`; un predicate `canApplyState` evite que plusieurs notes modifient les memes objets.
- Le rendu est deterministe: positions calculees depuis `currentSongPosition`, `Note.SongPosition` et `ApproachDuration`, pas depuis `GameTime`.
- Le fichier contient des patterns reutilisables:
  - calcul de progression normalisee dans une fenetre `[SongPosition - ApproachDuration, SongPosition]`;
  - detection de rewind avec epsilon;
  - reset de triggers one-shot au rewind ou avant la fenetre;
  - animation event-like `jump/land` declenchee une seule fois;
  - forcer un state d'animation, avec re-entree speciale pour `land`;
  - arc sine deterministe entre deux positions;
  - duree d'approche basee sur proximite inner/outer;
  - phases/handoff via progressions normalisees.
- `SeeSaw.cs` duplique aussi une partie de la logique temporelle: selection de la note conductrice, rewind scene, reconstruction de l'etat stable.

## Fichiers a modifier
- Ajouter `Gameplay/Visual Notes/RhythmVisualUtils.cs`.
- Refactorer `Gameplay/Visual Notes/SeeSawVisualNote.cs` pour utiliser les helpers.
- Optionnel si utile et sans gros changement: ajuster `Scenes/SeeSaw.cs` pour utiliser un helper de fenetre active ou de rewind, mais garder les helpers specififiques See-Saw dans la scene si cela evite un refactor trop large.

## Design propose

### 1. `RhythmVisualUtils` statique
Creer une classe statique sans namespace, coherente avec les fichiers existants, avec des helpers generiques et sans dependance a See-Saw.

API proposee:
- `public const double DefaultRewindThreshold = 0.001;`
- `public static bool HasRewound(double currentSongPosition, double lastSongPosition, double threshold = DefaultRewindThreshold)`
- `public static double GetApproachStart(double hitTime, double approachDuration)`
- `public static bool IsBeforeApproach(double currentSongPosition, double hitTime, double approachDuration)`
- `public static bool IsAtOrAfterHit(double currentSongPosition, double hitTime)`
- `public static float GetApproachProgress(double currentSongPosition, double hitTime, double approachDuration)`
  - Retourne la progression non clamp ou clamp? Preferer clamp `[0, 1]` pour eviter les surprises dans les nouveaux fichiers, et ajouter si necessaire `GetUnclampedApproachProgress`.
- `public static float GetPhaseProgress(float progression, float phaseStart, float phaseEnd)`
  - Convertit une progression globale vers une progression locale clamp `[0, 1]`.
- `public static bool CanApplyState(Func<bool> canApplyState)`
- `public static float SineArcHeight(float progression)`
- `public static Vector2 SineArc(Vector2 from, Vector2 to, float height, float progression)`
- `public static void ApplySineArc(GameObject target, Vector2 from, Vector2 to, float height, float progression)`
- `public static bool IsNearer(Vector2 value, Vector2 a, Vector2 b)`
- `public static double ApproachDurationByNearestReference(double beatDuration, Vector2 from, Vector2 shortReference, Vector2 longReference, double shortBeats, double longBeats)`

### 2. `RhythmVisualTimeline` petit struct optionnel
Pour eviter de repeter trois doubles dans chaque VisualNote:
- `public readonly struct RhythmVisualTimeline`
  - `HitTime`, `ApproachDuration`, `StartTime`.
  - `ContainsApproach(current)`, `IsComplete(current)`, `GetProgress(current)`.

Ce struct est utile si le refactor de `SeeSawVisualNote.Update` devient plus lisible. Sinon, garder uniquement la classe statique pour limiter l'ampleur.

### 3. `RhythmAnimationTriggerGate<T>` optionnel
Extraire proprement les one-shot flags sans coupler a SeeSaw:
- `public sealed class RhythmAnimationTriggerGate<T>`
  - `bool TryTrigger(T key, string triggerName)` ou `bool TryTrigger(T key, RhythmVisualTrigger trigger)`.
  - `void Reset()`.

Pour ce projet, un helper plus simple peut suffire:
- `public static bool TrySetTrigger(ref bool triggered)`
  - Si false, met true et retourne true; sinon retourne false.

Je privilegie `TrySetTrigger(ref bool)` dans une premiere passe: c'est minimal et evite de sur-abstraire.

### 4. Helper animation state machine
Ajouter des helpers generiques dans `RhythmVisualUtils`:
- `public static void ForceAnimationState(AnimationStateMachine stateMachine, string stateName, string reenterViaState = null)`
  - Si `stateMachine` null: return.
  - Si state courant identique et pas de reentry: return.
  - Si state courant identique et `reenterViaState != null`: force d'abord le state de transition, puis le state demande.
- `public static void ForceAnimationState<TKey>(IReadOnlyDictionary<TKey, AnimationStateMachine> states, TKey key, string stateName, string reenterViaState = null)`

Cela remplace la logique specifique de `ForceAnimationState` dans SeeSaw, en gardant `LandState` re-enter via `JumpState` cote appel.

## Refactor SeeSawVisualNote
1. Remplacer les calculs dans `Update` par les helpers:
   - `RhythmVisualUtils.HasRewound(...)`.
   - `RhythmVisualUtils.CanApplyState(_canApplyState)`.
   - `RhythmVisualUtils.IsBeforeApproach(...)`.
   - `RhythmVisualUtils.GetApproachProgress(...)`.
   - `RhythmVisualUtils.IsAtOrAfterHit(...)`.
2. Remplacer `ApplyJump` par `RhythmVisualUtils.ApplySineArc(...)` ou `SineArc(...)`.
3. Remplacer `SeeSawJumpPath.GetApproachDuration` par un appel a `ApproachDurationByNearestReference`, tout en gardant `SeeSawJumpPath` dans le meme fichier si sa semantique reste See-Saw.
4. Remplacer `ForceAnimationState` par le helper generique, puis supprimer la methode privee si elle ne sert plus.
5. Remplacer les triggers one-shot dans `StartJump` et `Land` par `RhythmVisualUtils.TrySetTrigger(ref flag)` pour reduire le bruit sans changer la logique.
6. Supprimer les helpers morts `EaseOutCubic` et `SmoothStep` s'ils restent inutilises apres verification.
7. Garder dans `SeeSawVisualNote` ce qui est vraiment specifique:
   - noms d'etats `jump/fall/land`;
   - `SeeSawJumper`, `SeeSawCounterJump`, `SeeSawJumpPath`;
   - rotations du beam;
   - camera big leap;
   - choreography counter/main jump.

## Points d'attention
- Ne pas changer la signature publique principale de `SeeSawVisualNote` tant que `SeeSaw.cs` l'utilise.
- Verifier `ApproachDuration` si elle peut valoir 0: le helper doit eviter division par zero, probablement retourner `1f` quand `current >= hitTime`, sinon `0f`.
- Garder les progressions clamp pour l'arc afin d'eviter les depassements si le manager appelle legerement hors fenetre.
- `Single.Lerp` existe probablement via .NET; ne pas y toucher sauf erreur de build.
- `SceneCamera` peut etre null dans certaines surcharges; `ApplyCameraJump` devrait rester protege si `isBigLeap` implique une camera non null, ou ajouter un guard minimal `_sceneCamera == null`.

## Verification
- Lancer `dotnet build` depuis la racine du projet.
- Si le build revele des erreurs venant de l'API `AnimationStateMachine`, ajuster uniquement le helper concerne.
- Comparer rapidement les usages de `SeeSawVisualNote.GetApproachDuration` et `SeeSawJumpPath.GetApproachDuration` pour s'assurer que les timings restent identiques.

## Resultat attendu
- Un nouveau `RhythmVisualUtils.cs` concentre les patterns de timing/determinisme/reutilisation.
- `SeeSawVisualNote.cs` reste fonctionnel mais plus lisible, avec moins de logique generique inline.
- Les prochains fichiers de VisualNote peuvent partir de ces helpers au lieu de recopier le spaghetti See-Saw.
