# Refactor Plan: Elements/Editor Notes

## Goal
Make adding an editor note require one obvious registration point instead of creating a definition provider and separately wiring an options panel. Keep behavior unchanged for existing RhythmInput and SeeSaw notes.

## Current Issues
- `EditorNoteDefinitions` owns note definitions through `IEditorNoteDefinitionProvider`.
- `EditorNoteOptionsPanels` owns options panels through a separate `Dictionary<EditorNoteKind, IEditorNoteOptionsPanel>`.
- A developer adding a note must know to update multiple registries and potentially multiple files.
- `BeatmapEditorElement` manually coordinates definition lookup and panel lookup.
- Note-specific code is split in a way that does not communicate the intended extension pattern.

## Proposed Shape
Introduce a single note module/provider abstraction that contains everything the editor needs for one note kind:

```csharp
public interface IEditorNoteProvider
{
    EditorNoteDefinition Definition { get; }
    IEditorNoteOptionsPanel OptionsPanel { get; }
}
```

Add a small base class to make simple notes concise:

```csharp
public abstract class EditorNoteProvider : IEditorNoteProvider
{
    public abstract EditorNoteDefinition Definition { get; }
    public virtual IEditorNoteOptionsPanel OptionsPanel => null;
}
```

Then each note has one entry file:

```csharp
public sealed class RhythmInputEditorNote : EditorNoteProvider
{
    public override EditorNoteDefinition Definition { get; } = new EditorNoteDefinitionBuilder(...).Build();
}
```

```csharp
public sealed class SeeSawEditorNote : EditorNoteProvider
{
    public override EditorNoteDefinition Definition { get; } = new EditorNoteDefinitionBuilder(...).Build();
    public override IEditorNoteOptionsPanel OptionsPanel { get; } = new SeeSawEditorNoteOptionsPanel();
}
```

## Implementation Steps
1. Replace `IEditorNoteDefinitionProvider` with `IEditorNoteProvider` or change the existing interface to expose both `Definition` and optional `OptionsPanel`.
2. Add `EditorNoteProvider` base class for optional panel defaulting to `null`.
3. Refactor `RhythmInputEditorNote` and `SeeSawEditorNote` to expose `Definition` properties instead of `Create()` methods.
4. Collapse `EditorNoteOptionsPanels` into `EditorNoteDefinitions` or rename the registry to a broader name such as `EditorNoteRegistry`.
5. Keep public call sites simple by exposing:
   - `All` definitions for selection UI.
   - `Get(EditorNoteKind kind)`.
   - `FromChartNote(ChartNote note)`.
   - `TryGetOptionsPanel(EditorNoteKind kind, out IEditorNoteOptionsPanel panel)`.
   - `FindVariantIndex(EditorNoteDefinition definition, ChartNote note)`.
6. Update `BeatmapEditorElement` to use the unified registry instead of `EditorNoteOptionsPanels`.
7. Delete the separate `EditorNoteOptionsPanels` class once all references are gone.
8. Preserve the existing special SeeSaw variant matching behavior inside `FindVariantIndex` for now to minimize behavior risk.

## Minimal API Example After Refactor
Adding a simple note should look like this:

```csharp
public sealed class MyEditorNote : EditorNoteProvider
{
    public override EditorNoteDefinition Definition { get; } = new EditorNoteDefinitionBuilder(EditorNoteKind.MyNote, "My Note")
        .Occupies(beforeBeats: 1, afterBeats: 0.25)
        .HitWindow(beforeBeats: 0, afterBeats: 0.25)
        .Variant("Default")
        .Build();
}
```

Adding a configurable note should only add the panel in the same provider:

```csharp
public override IEditorNoteOptionsPanel OptionsPanel { get; } = new MyEditorNoteOptionsPanel();
```

Then the only required registry edit is adding `new MyEditorNote()` to the unified provider list.

## Validation
- Build the project after refactor.
- Smoke-check editor flows by code inspection if the app cannot be launched:
  - note selection still uses `All`.
  - creating a SeeSaw note opens its options panel.
  - creating a RhythmInput note works without options panel.
  - editing an existing configurable note still resolves its panel from the note kind.

## Non-Goals For This Pass
- Do not rewrite SeeSaw timing/state logic.
- Do not change chart serialization.
- Do not introduce reflection or automatic assembly scanning unless explicitly requested; an explicit provider list is simpler, safer, and predictable for this MonoGame project.
- Do not redesign the DevUI window system.
