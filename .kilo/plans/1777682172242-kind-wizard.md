# Plan: SeeSaw Opposite Actor Selection

## Goal
Add an editor dropdown that appears only when `Opposite` is selected, allowing the chart author to choose which pony moves to the side opposite their previous position. The non-selected pony stays in place. Update runtime visual/timeline behavior accordingly.

## Files To Change
- `Elements/Editor/Notes/SeeSaw/SeeSawEditorNoteOptionsPanel.cs`
- `Elements/Editor/Notes/SeeSaw/SeeSawAction.cs`
- `Scenes/SeeSaw.cs`
- `Elements/Editor/BeatmapEditorDocument.cs`
- Possibly `Gameplay/Visual Notes/SeeSawVisualNote.cs` only if current `SeeSawVisualNote` cannot represent Applejack as the main jumper cleanly after scene changes.

## Data Model
1. Add a persisted additional-data key in `SeeSawAction`, for example `opposite_jumper`.
2. Add an enum or small parser for the selected opposite jumper using existing `SeeSawJumper` values:
   - default: `RAINBOW_DASH` to preserve existing charts
   - alternate: `APPLEJACK`
3. Add helpers to `SeeSawAction`:
   - `GetOppositeJumper(IReadOnlyDictionary<string,string>)`
   - `SetOppositeJumper(IDictionary<string,string>, SeeSawJumper)`
   - instance property `OppositeJumper`
4. Ensure `FromAdditionnalData` returns an action carrying `OppositeJumper`, while non-opposite actions can ignore it.
5. Update `Apply(SeeSawEditorState state)`:
   - Existing default behavior remains Rainbow moves opposite Applejack.
   - If `OppositeJumper == APPLEJACK`, Applejack becomes opposite Rainbow and Rainbow stays where it was.

## Editor Panel
1. Build rows with a `List<DevUiWindowRow>` instead of a fixed array.
2. Keep the existing direction dropdown.
3. If base direction is `Opposite`, insert a dropdown after `TARGET`:
   - label: `JUMPER` or `OPPOSITE JUMPER`
   - options: `Rainbow Dash`, `Applejack`
   - selected index from `SeeSawAction.GetOppositeJumper(context.Note.AdditionnalData)`
   - on select: call `SeeSawAction.SetOppositeJumper` on `context.GetCurrentNote().AdditionnalData`
4. Keep Big Leap checkboxes unchanged.
5. Optionally remove the `opposite_jumper` key when switching away from `Opposite`; not required for behavior, but keeps note data cleaner.

## Runtime Scene Logic
1. Replace `CreateOppositeJumpVisual` with logic that can build either:
   - Rainbow main jump, Applejack stationary counter jump (current behavior)
   - Applejack main jump, Rainbow stationary counter jump (new behavior)
2. For Rainbow-selected opposite:
   - target is opposite of Applejack’s current stable side
   - Applejack remains at `applejackFromPos`
   - main jumper: `RAINBOW_DASH`
   - counter jumper: `APPLEJACK`
   - final beam rotation toward Rainbow
3. For Applejack-selected opposite:
   - target is opposite of Rainbow’s current stable side
   - Rainbow remains at `rainbowFromPos`
   - main jumper: `APPLEJACK`
   - counter jumper: `RAINBOW_DASH`
   - final beam rotation toward Applejack
4. Generalize `CreateRainbowDrivenVisual` or add a `CreateDrivenVisual` helper accepting:
   - `SeeSawJumper mainJumper`
   - `SeeSawJumpPath mainPath`
   - `SeeSawCounterJump? counterJump`
   - final beam rotation based on `mainJumper`
   - `isBigLeap`, `jumpMultiplier`, `counterJumpEndProgression`, `counterIsBigLeap`
5. Keep existing paired outer/inner actions using Rainbow as main jumper to avoid behavior changes outside `Opposite`.

## Timeline And Timing
1. Update stable target helpers in `Scenes/SeeSaw.cs`:
   - `GetRainbowTargetPosition`: for opposite, return fallback when Applejack is selected; otherwise existing opposite target.
   - `GetApplejackTargetPosition`: for opposite, return opposite of Rainbow’s current stable side when Applejack is selected; otherwise fallback.
2. Add helper `GetOppositeApplejackPosition(Vector2 currentRainbowPosition)`:
   - if Rainbow is outer, Applejack target is inner; otherwise outer.
3. Update `ApplyTimelineBaseState`, `GetRainbowPositionBefore`, and any replay code carefully so the current positions passed into target helpers reflect both actors’ pre-note stable state before either target is mutated.
4. Update `GetApproachDuration(Note note)`:
   - for opposite/Rainbow selected, keep existing timing based on Applejack counter phase plus Rainbow main phase.
   - for opposite/Applejack selected, calculate counter phase from Rainbow’s stationary/current path and main phase from Applejack’s path to its opposite target.
5. Update `BeatmapEditorDocument` contextual timing helpers:
   - `GetBeforeUsesOuterTiming`: opposite should use the selected counter/stationary actor’s current side for the first phase.
   - `GetAfterUsesOuterTiming`: opposite should use selected main actor’s target side.
   - `GetCounterUsesOuterTiming`: mirror runtime phase selection.
   - Ensure default Rainbow selection preserves current editor timing.

## Visual Note Changes
Prefer no structural changes in `SeeSawVisualNote` if `CreateDrivenVisual` can pass `SeeSawJumper.APPLEJACK` as the main jumper and a Rainbow counter jump. Verify:
- `_jumper` is already generic.
- `_counterJump` is already generic.
- beam target rotation is constructor-provided.
- camera big leap currently follows `_jumper`, which is correct for selected main jumper.
If Applejack main big leap should use a different animation policy than Rainbow, only then adjust `ApplyMainJump`; otherwise leave unchanged.

## Validation
1. Build the project with the existing solution/project command if available.
2. Manually inspect compile errors around `SeeSawAction` constructor/property changes.
3. Verify old charts without `opposite_jumper` still behave as Rainbow-opposite.
4. Verify editor panel only shows the new dropdown for base `Opposite`, including big-leap opposite variants.
5. Verify runtime stable state and in-flight visual for both choices:
   - Rainbow selected: Applejack stays, Rainbow moves opposite Applejack.
   - Applejack selected: Rainbow stays, Applejack moves opposite Rainbow.
