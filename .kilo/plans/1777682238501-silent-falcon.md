# Plan: Fix SeeSaw Opposite Modes And Add Both

## Goal
Fix the current `Opposite` behavior so the non-selected pony really stays in place, then add a third editor/runtime option `Both` where Rainbow Dash and Applejack both jump to their own opposite side.

Assumption: `Both` means each pony toggles from its own previous side, not "opposite of the other pony". Example: Rainbow inner -> outer, Applejack outer -> inner.

## Current Problems Found
- `SeeSawAction.OppositeJumper` uses `SeeSawJumper`, which cannot represent `Both`.
- `opposite_jumper` defaults to Rainbow Dash, which preserves old charts, but the current runtime implementation still creates a stationary counter jump for the non-selected pony.
- Because the non-selected pony still receives a `SeeSawCounterJump`, single-jumper `Opposite` notes can visually make both ponies jump even though the intended behavior is that the non-selected pony stays in place.
- The example final note starts with Applejack outer and Rainbow inner. With default Rainbow-selected `Opposite`, Rainbow's target is already inner, and Applejack is only a stationary counter jump. This explains the confusing "nothing, then both normal jumps" behavior: the note is not expressing the desired "both swap sides" action.
- `GetBeamTargetRotation()` currently assumes every `Opposite` resolves toward a single selected `SeeSawJumper`; `Both` will need an explicit final beam decision.
- Editor timing helpers in `BeatmapEditorDocument` are tied to one selected main actor/counter actor. `Both` needs a deterministic timing rule.

## Data Model
1. Add a new enum near `SeeSawAction`, for example:
   - `SeeSawOppositeMode.RainbowDash`
   - `SeeSawOppositeMode.Applejack`
   - `SeeSawOppositeMode.Both`
2. Keep the persisted key as `opposite_jumper` for compatibility.
3. Supported persisted values:
   - missing or `rainbow_dash` -> `RainbowDash`
   - `applejack` -> `Applejack`
   - `both` -> `Both`
4. Replace or supplement `OppositeJumper` with `OppositeMode`.
5. Keep helper compatibility where useful:
   - `GetOppositeMode(IReadOnlyDictionary<string,string>)`
   - `SetOppositeMode(IDictionary<string,string>, SeeSawOppositeMode)`
   - Optional: keep `GetOppositeJumper`/`SetOppositeJumper` only as wrappers if other code still needs them.
6. Update `SeeSawAction.Apply()`:
   - `RainbowDash`: Rainbow becomes `!state.ApplejackIsOuter`; Applejack unchanged.
   - `Applejack`: Applejack becomes `!state.RainbowIsOuter`; Rainbow unchanged.
   - `Both`: Rainbow becomes `!state.RainbowIsOuter`; Applejack becomes `!state.ApplejackIsOuter`.

## Editor Panel
1. Change the dropdown label/options for `Opposite`:
   - `Rainbow Dash`
   - `Applejack`
   - `Both`
2. Use `SeeSawAction.GetOppositeMode(...)` for selected index.
3. Use `SeeSawAction.SetOppositeMode(...)` on selection.
4. Preserve existing conditional visibility: only show this dropdown when base direction is `Opposite`.
5. Keep the existing dropdown key `see_saw_opposite_jumper` unless renaming is worth it; the UI label can still say `OPPOSITE MODE`.

## Runtime Visual Logic
1. Stop using a stationary counter jump for single-jumper `Opposite` modes.
2. Add explicit helpers in `Scenes/SeeSaw.cs`:
   - `CreateRainbowOppositeVisual(...)`: one main jumper, Rainbow Dash, no counter jump.
   - `CreateApplejackOppositeVisual(...)`: one main jumper, Applejack, no counter jump.
   - `CreateBothOppositeVisual(...)`: both ponies jump to their own toggled side.
3. For single-jumper modes:
   - target position is computed from stable pre-note state.
   - non-selected pony is not passed as `SeeSawCounterJump`; it remains under `ApplyTimelineBaseState()` until the hit resolves.
   - big leap flag follows the selected jumping pony:
     - Rainbow-selected uses `action.IsBigLeap` / `big_leap_rainbow_dash`.
     - Applejack-selected uses `action.HasBigCounterJump` / `big_leap_applejack` as Applejack's main jump big-leap flag.
   - approach duration is based on the selected pony's path and selected pony's big-leap flag.
4. For `Both`:
   - Rainbow target: `rainbowOuterPos` if currently inner, else `rainbowInnerPos`.
   - Applejack target: `applejackOuterPos` if currently inner/exit, else `applejackInnerPos`.
   - Both should animate during the same approach window, not counter-then-main, because the requested behavior is “les deux sautent”.
   - Use per-pony big leap flags independently.
5. Implementing `Both` visual:
   - Preferred minimal change: extend `SeeSawVisualNote` to support an optional second simultaneous jump path, separate from `SeeSawCounterJump`.
   - New optional data could be `SeeSawSecondaryJump? _simultaneousJump`, with its own `isBigLeap` flag and target rotation if needed.
   - `ApplyCompletedState()` lands both actors and snaps both to final positions.
   - `ApplyMainJump()` continues to drive the primary jumper.
   - Add `ApplySimultaneousJump()` that uses the same active phase/progression as the primary jump, not the counter phase.
   - Keep existing counter behavior for normal Outer/Inner paired jumps.
6. Beam final rotation for `Both`:
   - Choose a deterministic rule. Recommended: keep final beam rotation toward Rainbow Dash, matching normal notes and old runtime behavior.
   - If desired later, this can become another visual rule, but it should not block this feature.

## Runtime Stable Timeline
1. Update `GetRainbowTargetPosition(...)`:
   - `RainbowDash`: `GetOppositeRainbowPosition(currentApplejackPosition)`.
   - `Applejack`: fallback.
   - `Both`: toggle Rainbow from its current stable position.
2. Update `GetApplejackTargetPosition(...)`:
   - `RainbowDash`: fallback.
   - `Applejack`: `GetOppositeApplejackPosition(currentRainbowPosition)`.
   - `Both`: toggle Applejack from its current stable position.
3. Add helpers:
   - `GetToggledRainbowPosition(Vector2 currentRainbowPosition)`.
   - `GetToggledApplejackPosition(Vector2 currentApplejackPosition)`.
4. Keep replay order safe: always compute next Rainbow and next Applejack from old pre-note positions before assigning either.
5. Update `GetBeamTargetRotation(...)`:
   - single Rainbow -> Rainbow rotation.
   - single Applejack -> Applejack rotation.
   - Both -> Rainbow rotation (recommended deterministic default).

## Runtime Approach Duration
1. Replace the current `Opposite` branch with mode-specific logic.
2. For `RainbowDash`:
   - duration from Rainbow's current stable path to Rainbow's opposite target.
   - if Rainbow big leap: `2 * crotchet`.
3. For `Applejack`:
   - duration from Applejack's current stable path to Applejack's opposite target.
   - if Applejack big leap: `2 * crotchet`.
4. For `Both`:
   - compute Rainbow duration from Rainbow path and Rainbow big-leap flag.
   - compute Applejack duration from Applejack path and Applejack big-leap flag.
   - approach duration should be the max of both durations so both can start together and land at the same hit time.
5. Keep normal Outer/Inner paired timing unchanged unless compile/runtime checks expose drift.

## Editor Timing
1. Update `BeatmapEditorDocument` helpers to use `OppositeMode`.
2. `GetBeforeUsesOuterTiming(...)`:
   - `RainbowDash`: Rainbow's current side.
   - `Applejack`: Applejack's current side.
   - `Both`: likely the longer of both actors' needed timings cannot be represented by one bool; add a special mode path if needed.
3. `GetAfterUsesOuterTiming(...)`:
   - `RainbowDash`: Rainbow target side after applying action.
   - `Applejack`: Applejack target side after applying action.
   - `Both`: target side for the actor whose timing dominates, or a special max-duration helper.
4. Because the existing `IEditorNoteTiming` API only accepts booleans, the cleanest fix is likely to update `BeatmapEditorDocument.GetContextualStart/GetContextualHitWindowStart/GetContextualHitWindowEnd` to special-case SeeSaw `OppositeMode.Both` and compute exact seconds there if the existing timing API cannot express max(two independent jumps).
5. Preserve old charts: missing `opposite_jumper` keeps old Rainbow mode, but now with the corrected “only Rainbow jumps” visual.
