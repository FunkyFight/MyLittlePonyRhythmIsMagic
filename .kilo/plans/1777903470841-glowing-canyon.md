# Plan de correction Seapony Parade

## Constats

- `SeaponyVisualNote.handleRoll` force l'etat `roll` avant de verifier si le joueur est autorise a roller. Pour le seapony 1, un miss ou une note pas encore autorisee peut donc afficher brievement l'etat `roll` sans vraie rotation.
- Le roll joueur demarre depuis `_rollStartSongPosition = currentSongPosition` et `_rollStartRotation = _seaPony.Rotation`. Le resultat depend donc de la frame ou l'input est traite et de l'etat visuel courant, pas uniquement de la position musicale.
- `SeaPonyParade.createReactions` applique directement `ForceState("swim")` sur le joueur au moment exact de l'evenement d'input. Un hit early/late change donc le timing visuel de l'animation.
- Plusieurs `SeaponyVisualNote` d'un meme poney peuvent encore etre vivantes en meme temps via `LookBehindSeconds` et leurs time windows. Les anciennes notes peuvent executer leur cleanup (`idle`, reset rotation) dans la meme frame qu'une note plus pertinente, ce qui cree des last-writer-wins et des flickers.
- Les appels directs a `ForceState` dans les chemins repétés (`swim_anticipation`, `idle`, `swim`) risquent de re-enter le meme state chaque frame. `RhythmVisualUtils.ForceAnimationState` existe deja pour eviter ca.
- Les subscriptions faites a `BeatmapStarted` / `NoteReactedWithNote` ne sont pas nettoyees dans `OnUnload` et ne gardent pas de reference au `ChartPlayer` courant. Un reload de scene peut multiplier les handlers, les SFX et les mutations d'etat.
- Les SFX de cues sont joues depuis chaque visual note par poney. Ce n'est pas la source principale des flickers, mais ca rend les side effects dependant du nombre de managers actifs et des subscriptions.

## Objectifs

- Une frame donnee doit avoir un seul proprietaire visuel par seapony.
- Pour le seapony 1, une reaction input doit enregistrer un resultat, pas changer directement son animation de parade.
- Les animations doivent etre calculees depuis `currentSongPosition`, `Note.SongPosition` et l'etat de reaction de la note, sans utiliser la frame de reception comme origine d'animation.
- Les changements doivent rester petits et lisibles, sans porter toute l'architecture de `SeeSawScene`.

## Plan d'implementation

1. Ajouter un ownership minimal dans `SeaPonyParade`.

- Ajouter `private Note[] _drivingSeaPonyNotes = new Note[SeaPonyCount];`.
- Au debut de `updateSeaPonies`, calculer pour chaque poney la note qui a le droit de muter son `GameObject` cette frame.
- Regle simple proposee : choisir la derniere note seapony active pour ce poney dans la timeline, avec priorite a la phase post-hit active, puis a l'approche la plus proche du hit.
- Passer a chaque `SeaponyVisualNote` un `Func<int, Note, bool>` ou un `Func<bool>` de type `canApplyStateProvider`, reutilisable avec `RhythmVisualUtils.CanApplyState`.
- Dans `SeaponyVisualNote.Update`, mettre a jour `base.Update` et les flags internes, mais ne pas muter animation/rotation/SFX si la note n'est pas owner.

2. Remplacer les mutations d'animation directes de l'event input.

- Dans `createReactions`, pour `seapony_parade_swim`, ne plus appeler `_seaPoniesAnimationStates[1].ForceState("swim")`.
- Garder uniquement l'enregistrement du resultat necessaire aux visuels, par exemple via deux sets : `_successfulReactionNotes` et `_authorizedRollNotes`, ou un dictionnaire `Dictionary<Note, NoteReactionResult>`.
- Continuer a jouer le SFX de feedback si souhaite, car c'est un feedback input et non l'etat visuel de parade.
- Pour `MISS`, retirer/mettre a jour explicitement l'etat reaction de la note.

3. Rendre le roll joueur deterministe.

- Dans `handleRoll`, verifier l'autorisation avant `ForceAnimationState("roll")` pour `seaPonyIndex == 1`.
- Supprimer le demarrage base sur `currentSongPosition` comme origine variable. Utiliser une origine fixe : `rollStartSongPosition = Note.SongPosition`, `rollEndSongPosition = Note.SongPosition + rollDuration`.
- Pour la rotation de depart du joueur, eviter de lire la rotation courante mutable. Calculer la rotation de depart depuis la timeline de rolls, comme les autres poneys : `startRoll = _rollTargetRotation - 90f` ou une variante modulo stable.
- Si une reaction valide arrive apres le hit, l'animation doit se placer directement a la progression correspondant a `currentSongPosition`, pas allonger/raccourcir le roll.
- A la sortie de fenetre, remettre aussi l'animation du joueur dans un state stable (`idle` sauf si un autre owner actif prend la main), pas seulement `Rotation = 0`.

4. Rendre le swim joueur deterministe.

- `handleSwim` doit garder `swim_anticipation` pendant l'approche tant que la note n'est pas resolue.
- Au hit ou apres le hit, pour `seaPonyIndex == 1`, choisir l'etat depuis le resultat enregistre : `swim` si succes, `idle`/pas de mutation si miss.
- Ne pas basculer en `swim` avant `Note.SongPosition` juste parce que l'input est early.
- Utiliser `RhythmVisualUtils.ForceAnimationState` au lieu de `ForceState` pour `swim_anticipation`, `swim`, `idle`.

5. Nettoyer les subscriptions et resets de timeline.

- Stocker le `ChartPlayer` abonne dans un champ `_reactionChartPlayer` et le handler dans une methode nommee, pour pouvoir faire `-=` avant de re-s'abonner.
- Dans `createReactions`, desabonner l'ancien chart player avant d'abonner le nouveau.
- Dans `OnUnload`, desabonner `BeatmapStarted` et `NoteReactedWithNote`.
- Lors d'un nouveau beatmap ou d'un rebuild playback, vider les sets/dictionnaires de reactions, remettre `_drivingSeaPonyNotes` a `null`, reset rotations/states de base.

6. Reduire les side effects multiples de SFX.

- Pour les cues automatiques (`Bubble.wav`, `BubbleHeavy.wav`), designer un seul owner de SFX, par exemple `seaPonyIndex == 0` ou un helper scene-level avec `HashSet<(Note, string, double)>`.
- Garder les SFX input (`BubbleSwim.wav`) dans le handler de reaction si le feedback immediat est voulu.
- Cette etape peut etre separee si le but immediat est uniquement le flicker, mais elle contribue au determinisme des side effects.

## Verification prevue

- Build du projet avec `dotnet build`.
- Test manuel dans l'editeur : seek avant/apres une sequence swim puis roll, verifier que le seapony 1 ne reste pas en sprite `roll` apres un miss.
- Test manuel hit early/hit late : l'etat `swim` et la rotation du roll doivent s'aligner sur la timeline musicale, pas sur la frame d'input.
- Test manuel avec rolls consecutifs : pas d'une-frame idle/roll parasite aux frontieres de notes.
- Test reload/rebuild playback : pas de SFX doubles ni de handlers accumules.

## Priorite conseillee

1. Fix roll joueur : autorisation avant `roll`, origine temporelle fixe, cleanup state.
2. Fix event swim : l'event enregistre le resultat, le visuel applique au temps musical.
3. Ajouter ownership par poney pour supprimer les last-writer-wins.
4. Nettoyer subscriptions.
5. Centraliser ou limiter les SFX automatiques.
