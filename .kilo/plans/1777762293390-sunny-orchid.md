# Plan: corriger le crash au changement de musique

## Contexte observe
- `SeaPonyParade.cs` est la scene active initialisee dans `Game1.Initialize()`.
- Le changement de musique dans l'editeur passe par `BeatmapEditorElement.SelectSongRelative()` puis `RebuildPlayback(false)`.
- `RebuildPlayback()` appelle `_beatmapPlayer.StartBeatmapPaused(...)`, qui dispose le conducteur courant, reconstruit `Conductor`/`ChartPlayer`, puis invoque `BeatmapStarted`.
- `SeaPonyParade` n'est pas directement abonne a `BeatmapStarted`, mais son `Update()` et `Draw()` supposent que `_infiniteScrollBg` et les animation states sont toujours valides.
- `SeaPonyParade` contient aussi un `catch(Exception e){}` silencieux dans la mise a jour des sprites, ce qui masque une erreur potentielle pendant le changement d'etat.
- `SeeSawScene` ajoute un handler anonyme a `GLOBALS.beatmapPlayer.Conductor.BeatChanged` a chaque `BeatmapStarted` sans conserver/debrancher le handler. Ce n'est probablement pas le crash direct de `SeaPonyParade`, mais c'est un risque reel lors des reconstructions de musique.

## Hypothese principale
Le crash vient d'une exception non protegee pendant une frame de reconstruction du playback ou de scene: objet visuel null/non initialise, sprite remplace par un type inattendu, ou event d'ancien/new conductor declenche pendant que les references globales changent. Le `catch` silencieux dans `SeaPonyParade` empeche de voir la vraie cause si elle concerne les sprites.

## Plan d'implementation
1. Nettoyer `SeaPonyParade.cs` sans changer son comportement visuel:
   - Supprimer `using System.Reflection.Metadata;` inutilise.
   - Remplacer les nombres magiques `4` par une constante `SeaPonyCount` et initialiser les tableaux avec cette constante.
   - Ajouter des guards null dans `Update()`/`Draw()` pour `_infiniteScrollBg` et les animation states, afin que la scene ne crash pas si une frame arrive avant/apres une reconstruction incomplete.
   - Dans `updateSeaPonies`, verifier `_seaPoniesAnimationStates[i] != null` avant `Update()`.
   - Dans le global update des animations, remplacer le cast `AnimatedSprite asp = (AnimatedSprite) seaPony.sprite;` + `catch` silencieux par un pattern check `if (seaPony.sprite is AnimatedSprite animatedSprite) animatedSprite.Update(time);`.

2. Ajouter une observation utile si necessaire:
   - Si le crash persiste apres les guards, ajouter temporairement/logiquement un message console dans les points de rebuild (`BeatmapEditorElement.RebuildPlayback`) ou dans les exceptions inattendues pour identifier l'objet exact.
   - Ne pas garder de logs verbeux permanents si la cause est corrigee.

3. Corriger le risque d'evenements dans `SeeSawScene` si le crash ou des callbacks fantomes continuent:
   - Remplacer les lambdas anonymes imbriquees par des handlers nommes stockes.
   - Desabonner l'ancien `Conductor.BeatChanged` avant de s'abonner au nouveau conductor.
   - Desabonner `BeatmapStarted`, `BeatChanged`, et `NoteReacted` dans `OnUnload()`.
   - Eviter de capturer `GLOBALS.beatmapPlayer.Conductor` directement dans un handler qui peut devenir obsolete apres `DisposeConductor()`.

4. Validation
   - Lancer `dotnet build`.
   - Demarrer l'editeur si possible.
   - Changer de musique plusieurs fois avec les raccourcis de l'editeur.
   - Verifier que la scene `SeaPonyParade` continue de se dessiner et que la timeline/editor playback se reconstruit sans crash.

## Fichiers probables a modifier
- `Scenes/SeaPonyParade.cs`
- Eventuellement `Scenes/SeeSaw.cs` si les abonnements d'evenements sont confirmes comme source du crash ou de callbacks persistants.

## Approche minimale recommandee
Commencer par `SeaPonyParade.cs` uniquement: guards null + suppression du cast avec exception silencieuse. C'est le changement le plus petit et le plus directement relie au fichier signale. N'etendre a `SeeSaw.cs` que si la reproduction montre encore un crash ou des callbacks multiples apres changement de musique.
