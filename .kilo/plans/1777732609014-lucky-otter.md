# Refactor Plan: `Scenes/SeeSaw/`

## Objectif
Refactorer le runtime See-Saw pour sortir de la classe monolithique `Scenes/SeeSaw.cs`, réduire les effets de bord entre Rainbow/Applejack/beam, et rendre le système maintenable selon SOLID. Le refactor doit préserver le comportement visible actuel, mais remplacer l'architecture fragile actuelle par des composants clairement séparés, documentés et testables.

## Constats
- `SeeSawScene` mélange scene graph, positions viewport, animation states, parsing de notes, timeline stable, création de visual notes, feedback gameplay, ownership des objets partagés et règles de timing.
- `SeeSawVisualNote` pilote directement plusieurs responsabilités: mouvement, beam, camera, animations, fail/land feedback, counter jump, simultaneous jump et big leap.
- La scène possède deux `ChartPlayer` différents, donc les `Note` gameplay et visual ne sont pas les mêmes instances. Le feedback doit matcher par données, ce qui est fragile.
- Les états métier sont représentés par des `Vector2` comparés directement (`inner`, `outer`, `exit`). C'est difficile à raisonner et facile à casser.
- Les timings See-Saw sont dupliqués entre runtime et éditeur (`SeeSawEditorNoteTiming`).
- `SetupFeedbacks()` subscribe à chaque `BeatmapStarted` sans unsubscribe, donc risque de handlers dupliqués.
- L'ownership `_drivingVisualNote` est une source de bugs: un feedback post-hit peut devoir toucher Rainbow alors que la note n'a plus le droit de muter les GameObjects.

## Architecture Cible
Créer un dossier `Scenes/SeeSaw/` et déplacer progressivement la logique runtime dedans. Garder une classe scène mince qui orchestre seulement les composants.

### Fichiers Cibles
- `Scenes/SeeSaw/SeeSawScene.cs`
  - Remplace progressivement `Scenes/SeeSaw.cs` ou devient la nouvelle classe principale après migration.
  - Responsabilité: créer les GameObjects, appeler les services, appliquer les snapshots aux GameObjects, Update/Draw.

- `Scenes/SeeSaw/SeeSawLayout.cs`
  - Calcule et expose toutes les positions viewport et rotations initiales.
  - Contient `RainbowOuter`, `RainbowInner`, `ApplejackOuter`, `ApplejackInner`, `ApplejackExit`, `BeamBaseRotation`, `BeamTiltDegrees`.
  - Aucune logique de chart, aucun accès au gameplay.

- `Scenes/SeeSaw/SeeSawActorId.cs`
  - Remplacer ou encapsuler `SeeSawJumper` avec un enum runtime propre: `RainbowDash`, `Applejack`.
  - Si `SeeSawJumper` est utilisé par l'éditeur, éviter de le casser immédiatement; fournir des conversions.

- `Scenes/SeeSaw/SeeSawSide.cs`
  - Enum métier: `Outer`, `Inner`, `Exit`.
  - Interdit d'utiliser `Vector2 ==` pour décider d'un état métier.

- `Scenes/SeeSaw/SeeSawActorState.cs`
  - Position logique d'un acteur: actor + side.
  - Optionnellement `IsOnBoard` ou `CanJump` si nécessaire.

- `Scenes/SeeSaw/SeeSawTimelineState.cs`
  - Snapshot pur: sides/positions finales de Rainbow, Applejack, beam rotation.
  - Ne contient pas de `GameObject`.

- `Scenes/SeeSaw/SeeSawTimeline.cs`
  - Service pur qui rejoue les actions de chart jusqu'à `songPosition`.
  - Entrée: notes, actions, état initial.
  - Sortie: `SeeSawTimelineState`.
  - Remplace `ApplyTimelineBaseState`, `GetRainbowPositionBefore`, `GetApplejackPositionBefore`, `GetRainbowTargetPosition`, `GetApplejackTargetPosition`, `GetBeamRotationBefore`.

- `Scenes/SeeSaw/SeeSawActionReader.cs`
  - Lit `Note.AdditionnalData` et retourne un `SeeSawAction` runtime valide.
  - Devient le seul endroit runtime qui connaît `ActionDataKey`.
  - Documenter le fait que l'éditeur est la source des valeurs sérialisées.

- `Scenes/SeeSaw/SeeSawTiming.cs`
  - Centralise les constantes et fonctions de timing runtime: inner/outer beats, big leap duration, counter land progression, land trigger progression, hit ownership grace.
  - Doit être aligné explicitement avec `SeeSawEditorNoteTiming`.
  - À terme, faire utiliser les mêmes constantes côté éditeur pour supprimer la divergence.

- `Scenes/SeeSaw/SeeSawMotionPlan.cs`
  - Objet immutable qui décrit une visual note: phases, paths, actors, beam rotations, camera behavior.
  - Remplace les multiples constructeurs et flags de `SeeSawVisualNote`.

- `Scenes/SeeSaw/SeeSawMotionPlanner.cs`
  - Convertit une note + états timeline avant/après en `SeeSawMotionPlan`.
  - Contient les règles Outer/Inner/Opposite/Exit/BigLeap.
  - Remplace `CreateVisualNote`, `CreatePairedJumpVisual`, `CreateOppositeJumpVisual`, `CreateRainbowOppositeVisual`, etc.

- `Scenes/SeeSaw/SeeSawVisualNote.cs`
  - Nouvelle version ou déplacement de l'existant.
  - Responsabilité unique: évaluer un `SeeSawMotionPlan` à un temps donné et appliquer uniquement le mouvement/camera/beam prévu par ce plan.
  - Ne doit pas gérer le feedback gameplay `fail` directement.

- `Scenes/SeeSaw/SeeSawFeedbackController.cs`
  - Gère `ChartPlayer.NoteReacted`, miss automatique, wrong input, land/fail de Rainbow.
  - Ne modifie jamais Applejack.
  - Ne passe pas par l'ownership de visual note pour jouer `fail`.
  - Gère unsubscribe/resubscribe proprement.

- `Scenes/SeeSaw/SeeSawAnimationController.cs`
  - Crée et possède les `AnimationStateMachine` Rainbow/Applejack.
  - Expose des méthodes explicites: `PlayRainbowJump`, `PlayRainbowFall`, `PlayRainbowLand`, `PlayRainbowFail`, `PlayApplejackStartIdle`, etc.
  - Évite les strings dispersées (`"fail"`, `"land"`, `"jump"`).

- `Scenes/SeeSaw/SeeSawRuntimeActors.cs`
  - Regroupe les GameObjects et AnimationStateMachines.
  - Fournit un point d'accès typé au lieu de dictionnaires libres.

- `Scenes/SeeSaw/SeeSawVisualOwnership.cs`
  - Choisit la note visual qui peut appliquer les mouvements partagés.
  - Ne doit pas être utilisé pour le feedback gameplay.

## Principes SOLID Appliqués
- Single Responsibility: scène mince, timeline pure, planning motion, feedback, animations et layout séparés.
- Open/Closed: ajouter une action See-Saw doit se faire via `SeeSawMotionPlanner` et `SeeSawTimeline`, pas en modifiant la scène entière.
- Liskov: éviter les héritages complexes; garder `VisualNote` seulement pour l'intégration avec `VisualNoteManager`.
- Interface Segregation: petits services concrets avec API ciblée plutôt qu'un gros helper global.
- Dependency Inversion: `SeeSawScene` dépend de services runtime construits au load/start; les services purs ne dépendent pas de `GameObject`.

## Étapes D'Implémentation

### Étape 1: Préparation Sans Changement Comportemental
1. Créer `Scenes/SeeSaw/`.
2. Ajouter `SeeSawSide`, `SeeSawActorId`, `SeeSawTiming`, `SeeSawLayout`.
3. Garder `Scenes/SeeSaw.cs` fonctionnel pendant cette étape.
4. Remplacer seulement les constantes locales par `SeeSawTiming` quand c'est trivial.
5. Validation: build, lancement scène, aucune modification comportementale volontaire.

### Étape 2: Extraire La Timeline Stable
1. Créer `SeeSawTimelineState` avec états logiques (`SeeSawSide`) + positions calculées via `SeeSawLayout`.
2. Créer `SeeSawTimeline` qui applique les `SeeSawAction` dans l'ordre.
3. Remplacer `ApplyTimelineBaseState` par un appel au timeline service.
4. Remplacer `GetRainbowPositionBefore`, `GetApplejackPositionBefore`, `GetBeamRotationBefore` par `timeline.GetStateBefore(note)` ou équivalent.
5. Garder les anciennes méthodes temporairement si nécessaire, mais les marquer comme candidates à suppression dans l'étape suivante.
6. Validation: seek, rewind, première note, notes inner/outer/opposite/exit.

### Étape 3: Extraire Le Planning De Motion
1. Créer `SeeSawMotionPlan`, `SeeSawJumpPath` runtime et `SeeSawMotionPlanner`.
2. Déplacer toutes les règles de création de visual notes depuis la scène vers `SeeSawMotionPlanner`.
3. Convertir les fonctions `CreatePairedJumpVisual`, `CreateOppositeJumpVisual`, etc. en méthodes privées du planner ou en stratégies internes.
4. La scène ne doit plus contenir de switch complet sur `SeeSawDirection`.
5. Validation: comparer visuellement les trajectoires normales, big leap, counter big leap, opposite modes, exit.

### Étape 4: Réécrire `SeeSawVisualNote` Autour D'Un Plan
1. Remplacer les multiples constructeurs par un constructeur unique: `SeeSawVisualNote(Note logicalNote, SeeSawMotionPlan plan, SeeSawRuntimeActors actors, Func<bool> canApplyMotion)`.
2. Supprimer les surcharges anciennes si aucune référence externe ne les utilise; sinon garder une compatibilité temporaire minimale uniquement si nécessaire.
3. Séparer les phases: counter phase, main phase, simultaneous phase, beam phase, camera phase.
4. Supprimer tout feedback gameplay (`SnapLandRainbow`, fail) de la visual note, ou le limiter à un signal neutre sans toucher aux animations.
5. Validation: visual notes n'affectent que le mouvement prévu, pas le résultat input.

### Étape 5: Extraire Les Animations
1. Créer `SeeSawAnimationController`.
2. Déplacer `SetupAnimations` hors de la scène.
3. Remplacer les strings par constantes internes ou enum `SeeSawAnimationState`.
4. Fournir des méthodes haut niveau pour Rainbow et Applejack.
5. S'assurer que `PlayRainbowFail` ne modifie jamais Applejack.
6. Validation: idle bop, jump, fall, land, fail, start_idle Applejack.

### Étape 6: Isoler Le Feedback Gameplay
1. Créer `SeeSawFeedbackController`.
2. Il reçoit le `ChartPlayer` gameplay, un mapping visual si encore nécessaire, et `SeeSawAnimationController`.
3. Il unsubscribe proprement avant de resubscribe sur un nouveau beatmap.
4. Il gère trois cas explicitement:
   - hit correct: Rainbow land.
   - note auto-miss AFK: Rainbow fail.
   - wrong input/no active note: Rainbow fail sur la note actuellement pertinente si elle existe.
5. Il ne doit pas changer Applejack, beam, timeline stable ou visual ownership.
6. Validation: première note AFK, miss input trop tôt/trop tard, fail en dense chart, aucun changement Applejack pendant feedback fail.

### Étape 7: Nettoyer L'Ownership
1. Créer `SeeSawVisualOwnership`.
2. Garder l'ownership uniquement pour mouvement/beam/camera.
3. Interdire son usage dans feedback.
4. Documenter la règle: stable timeline d'abord, visual motion ensuite, feedback animation séparé.
5. Validation: overlapping notes, lookahead/lookbehind, rewind.

### Étape 8: Documentation Et Tests De Non-Régression
1. Ajouter XML docs aux nouveaux types publics/internal importants.
2. Ajouter commentaires de conception courts en haut de chaque service.
3. Si un framework de test existe ou peut être utilisé facilement, ajouter tests unitaires pour `SeeSawTimeline` et `SeeSawMotionPlanner`.
4. Sinon, ajouter une checklist manuelle dans le plan de validation ou un petit harness interne non invasif si accepté plus tard.

## Ordre De Migration Recommandé
1. Extraire layout/timing/types.
2. Extraire timeline pure.
3. Extraire motion planner.
4. Réduire `SeeSawVisualNote` à un plan unique.
5. Extraire animations.
6. Extraire feedback.
7. Déplacer/renommer la scène finale.

Cet ordre évite de toucher feedback et Applejack avant d'avoir séparé timeline et motion.

## Points D'Attention
- Ne pas réutiliser le feedback pour forcer l'ownership visual: c'est précisément ce qui a créé des collisions avec Applejack.
- Ne pas utiliser `Vector2 ==` comme état métier après extraction de la timeline.
- Ne pas multiplier les handlers `NoteReacted` / `BeatChanged` à chaque `BeatmapStarted`.
- Faire attention au namespace: les fichiers éditeur See-Saw sont sous `MLP_RiM.Elements.Editor`; le runtime See-Saw devrait probablement être dans un namespace runtime dédié si le projet en utilise déjà, sinon garder le style global actuel pour minimiser le diff.
- Le fichier actuel `Gameplay/Visual Notes/SeeSawVisualNote.cs` pourra être déplacé vers `Scenes/SeeSaw/` uniquement après avoir vérifié les références et les namespaces.

## Critères D'Acceptation
- `SeeSawScene` devient un orchestrateur lisible, idéalement moins de 250-350 lignes.
- Aucune méthode de scène ne contient la logique complète d'une action See-Saw.
- Le feedback `fail` de Rainbow ne peut plus modifier Applejack.
- Les états stables sont représentés par des enums/structs métier, pas par comparaison directe de positions.
- Une note AFK joue fail sur Rainbow.
- Une mauvaise input joue fail sur Rainbow sans casser Applejack.
- Les notes correctes jouent land sur Rainbow.
- Rewind/seek restent déterministes.
- Les actions Outer, Inner, Opposite, Exit, BigLeap, BigCounterJump fonctionnent comme avant.
- Build réussie avec `dotnet build "MLP_RiM.csproj" -p:OutDir="bin\\KiloVerify\\"` si le binaire principal est verrouillé par le debugger.

## Validation Manuelle Minimale
1. Lancer une chart See-Saw avec première note Outer normale.
2. Rester AFK sur la première note: Rainbow doit jouer fail, Applejack ne doit pas être déplacée par le feedback.
3. Réussir une note: Rainbow doit land.
4. Rater volontairement une input hors timing: Rainbow doit fail.
5. Tester Outer puis Inner: positions stables cohérentes.
6. Tester Opposite RainbowDash, Opposite Applejack, Opposite Both.
7. Tester Exit: Applejack sort sans beam/feedback parasite.
8. Tester BigLeap et BigCounterJump.
9. Scrub/rewind si l'éditeur le permet: pas de flags one-shot coincés.

## Risques
- Gros diff si tout est fait en une seule PR. Préférer des étapes compilables.
- Les classes éditeur utilisent `SeeSawAction`, `SeeSawDirection`, `SeeSawEditorState`; ne pas casser leur namespace ou API pendant le refactor runtime.
- Les bugs actuels autour du fail peuvent masquer d'autres problèmes dans les animations. Garder feedback isolé pour réduire le rayon d'impact.

## Recommandation Finale
Faire ce refactor en 3 à 5 commits logiques si possible:
1. Types/layout/timing + timeline pure.
2. Motion planner + visual note planifiée.
3. Animation controller + feedback controller.
4. Nettoyage `SeeSawScene` et suppression des anciennes méthodes.
5. Documentation et validation finale.
